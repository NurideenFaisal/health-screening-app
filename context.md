## Suggestion from previous
This is a major architectural milestone. By moving the logic into a single **Atomic Transaction** in an Edge Function, you've eliminated the risk of "partial successes" (like a clinic being created without an admin) and made the UI much cleaner.

Since you've verified the deployment and secrets, here is a pre-flight checklist to keep in mind before you hit that "Launch Outreach" button for the first time:

### 1. The "Super-Admin" Authorization Check
Inside your `launch-clinic/index.ts`, ensure you are validating the user's role using the **JWT** from the request header, not just checking if they are logged in. 
* **The Logic:** The function should decode the token, check the `public.profiles` table for the caller's ID, and verify `role === 'super-admin'`. 
* **Why:** This prevents a regular clinician from manually hitting the endpoint via Postman to create their own clinics.

### 2. Password Handling
Since you allowed for **Custom, UI-Generated, or Backend-Generated** passwords:
* Ensure the backend returns the *actual* password used in the final JSON response. 
* If the user types a custom password, the UI should probably have a "Show Password" toggle so they don't accidentally lock themselves out with a typo before the account is even created.

### 3. Future-Proofing the UI
Now that the backend is "one-shot," your UI's `loading` state is your best friend. 
* Since the backend is doing 4-5 heavy tasks (Validation -> Clinic Insert -> Auth Create -> Profile Insert), the request might take 2–4 seconds. 
* Make sure your "Creating..." spinner is prominent so the user doesn't refresh the page mid-transaction.

---

### End-to-End Test Plan
When you are ready for that first live payload, I recommend this **"Smoke Test"** sequence:

1.  **The Happy Path:** Create "Test Clinic A" with a unique code. Verify the clinic appears in the table and the user can log in.
2.  **The Duplicate Code Test:** Try creating "Test Clinic A" again with the *same* code. Verify the backend catches the uniqueness constraint and returns a clean error message to the UI.
3.  **The Rollback Test:** Intentionally send a malformed email (e.g., `not-an-email`) for the admin. Verify that the **Clinic** is *not* created in the database (proving your rollback worked).

# Transitioning from UI Builder to Supabase Backend

Moving from **volatile memory** (where everything disappears on refresh) to **persistent, relational storage** means rethinking how data is synchronized and identified. Your current "Canonical Form Schema" is well suited for this transition, but several changes are needed.

---

## 1. ID Generation Strategy
Currently:
- `_id = 200` and `genId()` create local strings like `f201`.

**The Change:**  
In a database environment, IDs must be globally unique to prevent collisions when multiple clinicians are creating forms.

**The Move:**  
Switch to **UUIDs** (Universally Unique Identifiers). Supabase can generate these automatically, or you can use `crypto.randomUUID()` in the browser before saving.

---

## 2. From Nested State to Relational Tables
Right now:
- `groups` array contains `fields` as a nested list.

Options with Supabase:
- **JSONB column type**: Save the entire structure as one blob (matches your current CFS v1 design).
- **Relational approach**:  
  - `form_groups`: Stores `id`, `label`, `color`, `sort_order`.  
  - `form_fields`: Stores `id`, `group_id` (foreign key), `type`, `label`, `logic_config`.

> **Note:** Since you mentioned using **JSONB** for clinical data orchestration, you might keep the *schema* in structured tables but store the *responses* in a flexible JSONB column to handle high‑volume, varying pediatric data.

---

## 3. Real-Time Sync & "Dirty" States
Currently:
- `isDirty` is a simple boolean.

In the backend version, decide on a **Persistence Strategy**:
- **Auto-save:** Every time `handleDragEnd` or `updateField` finishes, send a `PATCH` request to Supabase.
- **Manual Save:** Keep the "Publish" button, which sends the entire `groups` state to Supabase via `.upsert()` or RPC.

---

## 4. Logic Validation
Currently:
- `computed` fields and `conditions` run only in the browser’s `PreviewModal`.

**The Change:**  
Backend must not just trust the UI.

**The Move:**  
Use **Database Functions** (PostgreSQL) or **Edge Functions** to re‑validate:
- Required fields are not skipped.  
- Computed values are mathematically correct.  

This ensures integrity before committing to the clinical record.

---

## Implementation Mapping

| `INITIAL_GROUPS` constant | `.select()` query on mount to fetch the saved schema |
